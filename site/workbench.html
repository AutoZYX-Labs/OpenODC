厂家工作台解决什么问题

内部管理

  • 同一厂家往往同时维护多个车型、多个市场、多个 ADS 功能版本
  • ODC 信息需要连接产品、法规、功能安全、SOTIF、测试和售后追溯
  • 每次 SOP 或 OTA 都应保留当时的运行边界、证据来源和审核状态
  • 对外公开前，需要先区分可公开字段、内部字段和仍需核验字段

公开提交

  • 统一格式：使用 GB/T 45312—2025 的 144 项要素，减少自由文本解释空间
  • 证据绑定：每个关键要素都应关联手册、官方页面、运营规则或监管材料
  • 版本留痕：公开样例进入 GitHub 后，可通过提交历史追踪每次边界变化
  • 可审查发布：提交包进入 PR 流程后，由社区检查来源、术语和 L2/L3/L4 语义边界

推荐使用流程

  • 选择厂家或 Tier 1 身份，创建需要维护的 ADS 功能条目
  • 进入 ODC 编辑器，按 144 项国标元素补充允许条件、限制条件和退出行为
  • 为关键要素绑定官方手册、产品页、运营规则或监管材料
  • 在 SOP、OTA 或公开发布节点生成 PR 提交包
  • 通过 GitHub PR 接受公开审查，合并后进入 OpenODC 样例库
