为什么需要「厂家直填」

现状痛点

  • 每家 OEM / Tier 1 同时开发 5–30+ 个 ADS 功能（不同车型、不同等级、不同市场）
  • ODC 散落在 Confluence / 飞书 / Excel 里，每个项目组格式各异
  • 上市前法规审查、媒体测评、事故追溯，都需要翻回"当时的"ODC 声明
  • 对外发布的 ODC 文本无法机器读取，监管 / 第三方做横向对比成本高

厂家直填的价值

  • 统一格式：国标 GB/T 45312—2025 提供的标准化结构，自然成为内部所有项目的通用语言
  • 版本管理：从 SOP 开始，每次 OTA 都带一份 ODC 版本号；事后追溯直接定位
  • 协作编辑：架构 / 功能安全 / 法规 / 产品多角色共编，字段级权限
  • 一键公开：SOP 达成后，对外版本（脱敏后）自动推送 OpenODC；既解决"对外披露"又贡献行业标准化

正式版路线图

  • Phase 4.0（当前）：前端 localStorage 演示 — 验证工作流
  • Phase 4.1：账号体系（OEM 管理员 / 项目成员 / 只读观察员）+ Supabase 后端
  • Phase 4.2：功能组织结构（按车型 / 按等级 / 按市场筛选 + 标签）
  • Phase 4.3：版本历史 + diff（看每次 OTA 改了哪些要素）+ 字段级注释
  • Phase 4.4：签名发布 + GitHub App 自动创建 PR 至公开样例库
  • Phase 4.5：OEM SaaS 订阅 — 让项目长期可持续
