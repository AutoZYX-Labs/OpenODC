01 · ODC 是什么

运行设计条件（ODC）的工程定义

运行设计条件不是一句“支持高速”或“支持城区”就能概括，它需要把道路、交通、天气、目标物、数字信息、人和车的状态逐项讲清楚。

  • 系统可以在哪些道路、速度和交通环境下激活？
  • 遇到施工、积水、弱车道线时，应继续、抑制激活还是退出？
  • 这些边界来自车主手册、官方说明、运营规则，还是第三方推定？
对象ADS 功能、运营服务或具体市场版本 单位ODC 元素、参数范围和组合条件 证据来源类型、版本信息与核验日期

这一页先把 ODC 从抽象缩写拉回工程问题：公开资料到底有没有讲清楚系统边界。

概念区分

自动化等级与运行边界的语义区分

对象主要回答的问题与 ODC 的关系
L2 辅助驾驶驾驶员持续承担监管与接管责任ODC 描述功能可用条件，不能解释为脱离监管。
L3 自动驾驶系统在限定条件内承担动态驾驶任务ODC 必须与退出、接管和最小风险策略一起说明。
L4 无人驾驶系统在限定区域或运营规则内运行地理围栏、天气、道路和运营许可成为核心边界。

因此，OpenODC 不比较“谁更高级”，而是比较公开边界是否讲清楚、是否有证据支撑。

这页防止听众把 OpenODC 误解为排行榜。等级和边界是两个维度。

02 · 国标背景

GB/T 45312—2025 的 ODC 表达框架

标准将自动驾驶系统设计运行条件拆成结构化元素，并要求说明允许、不允许、元素关联关系，以及不满足条件时的退出行为。

§5.4.a设计运行条件元素
§5.4.b允许 / 不允许条件
§5.4.c元素关联关系
§5.5不允许条件下的退出行为
输入GB/T 45312 要素清单与条款要求 转换允许 / 不允许、关联关系与退出行为 输出可审阅、可比较、可导出的 ODCDocument
7ODC 要素类别
144国标元素
2025标准发布年份

这里讲标准价值：它不是给行业增加术语，而是让不同厂家、测试机构、监管和研究人员围绕同一张边界表沟通。

国标要素体系

结构化

03 · 行业缺口

公共 ODC 数据层的行业缺口

车主手册边界表述分散在注意事项、限制条件和警告中。
官方网页常以功能卖点呈现，参数和版本边界不总是完整。
运营规则L4 Robotaxi 的范围、时段、区域、许可与城市政策相关。
第三方测评能补充观察，但必须和官方证据区分开。

公共层缺失带来的结果

  • 消费者难以看懂不同系统的真实边界。
  • 研究人员难以构造统一对比样本。
  • 测试机构难以复用公开证据和版本信息。
  • 标准条文难以进入实际工具链。
统一数据契约样例、矩阵、导出文件共享同一套字段语义。 证据版本管理保留公开来源、核验日期、市场区域和软件版本。 社区审阅流程区分 community_reviewed、vendor_confirmed 与公开缺口。

这里从标准转向平台：标准提供语法，但公开资料仍然需要整理、核验和持续维护。

OpenODC 的定位

OpenODC 的证据化定位与适用边界

OpenODC 是一个开源 ODC 数据平台。除非记录标记为 vendor_confirmed，样例均为社区基于公开资料提取。

当前默认状态 community_reviewed

表示样例经过社区整理和公开资料复核，但不等同于厂家官方确认。

不做智驾能力排名

不替代厂家内部 ODC 文档系统

不替代 ISO 21448 / ISO 26262 安全论证

不把推定包装成官方披露

正式场合必须讲清定位和边界，项目可信度来自克制。

04 · 方法论

OpenODC 数据治理原则

01

标准优先

JSON Schema 是契约，网页、矩阵和导出文件都是派生视图。

02

证据优先

有来源的 30 项，比没有来源的 80 项更有价值。

03

语义优先

L2、L3、L4 可共用元素表，但责任和边界含义不能混淆。

04

版本优先

软件版本、市场区域、证据核验日期必须被保留。

Schema 校验字段、枚举值、证据引用和组合边界必须可机器检查。 证据审计官方、手册、政府、测评、推定来源不能混写。 版本追踪同一车型或服务在不同市场、版本下可形成独立记录。

这页总结 OpenODC 的工程原则。它是未来复用到其他项目的核心。

组合边界

组合边界：从单项 ODC 到场景耦合

ODC 表格会把真实道路拆成单项要素，但道路风险常常来自多个边界条件同时出现。

雨 / 雾 / 逆光
车道线磨损
施工改道
驾驶员分心
这不是四个独立格子，而是一个组合边界问题。

为什么要加入第二层

  • 消费者真正遇到的是组合条件，而不是表格单项。
  • 工程审阅需要知道哪些组合可能触发抑制或退出。
  • SOTIF 分析需要把组合条件转化为触发条件候选。

这是 OpenODC 相比普通 ODC 表格更有深度的地方，要讲透。

boundary_combinations[]

boundary_combinations[] 的字段语义与约束

01related_element_ids

哪些 ODC 要素同时出现。

02relation_to_odc

inside / outside / boundary / unknown。

03expected_response

抑制激活、触发退出、接管或未披露。

04evidence_level

官方、手册、运营规则或公开缺口。

重要边界

OpenODC 只记录公开资料可支撑的典型组合边界。它可以支持消费者提示、工程审阅和研究建模，但不能替代厂家完整 SOTIF 安全分析。

这一页要防止误读：组合边界是公共证据结构化，不是认证结论。

从 ODC 到触发条件候选

OpenODC 可以作为 SOTIF 和道路数据分析的入口

OpenODC 设计边界端

把公开资料中的 ODC 声明、证据来源和组合边界转成可审阅的数据结构。

DRIVEResearch 真实暴露端

用航测自然驾驶数据刻画边界条件在真实道路中的出现频次、参数分布和典型场景。

ROAM 异常结果端

记录 L4 Robotaxi 运营异常、越界表现、远程干预和结果证据，反向提示高价值边界组合。

三者合起来形成“声明边界—真实暴露—异常结果”的闭环：OpenODC 说明系统边界如何被声明，DRIVEResearch 说明这些边界条件在真实交通中多常见，ROAM 说明接近或越过边界之后会出现什么运营后果。
