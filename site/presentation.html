真正的问题

不是它能开到哪里

而是它不能在哪里开。

什么天气要退出？

施工、积水、车道线不清时怎么处理？

公开资料到底说明了多少？

用这个问题抓住所有受众。消费者、监管、媒体、工程师都能理解：强不强是一回事，边界清不清楚是另一回事。

L2 / L3 / L4

自动化等级不是运行边界

同样是 L2，不同市场、不同软件、不同手册里的可用条件可能完全不同。等级回答责任分配，ODC 回答运行条件。

L2驾驶员持续负责ODC 描述功能适用条件
L3系统在限定条件内接管退出与接管条件必须清楚
L4限定区域自动驾驶地理围栏、天气、运营规则成为核心

这一页防止听众把 OpenODC 误解为“L2 到 L4 排行榜”。说明 OpenODC 处理的是运行条件，而不是自动化等级本身。

ODC 的大白话

ODC 是一张边界清单

一项智驾或自动驾驶功能，被设计在什么道路、天气、交通参与者、人员状态和车辆状态下使用。

道路 设施 目标物 天气 数字信息 人员状态 车辆状态

不要陷入术语。把 ODC 解释成边界清单，后面再说它如何被标准化成 144 个要素。

从标准到数据

标准给了语法，行业缺少表格

GB/T 45312—2025 给出了 ODC 结构。OpenODC 把这套结构转成机器可读的 JSON Schema、分类目录和量化枚举。

标准 PDF
144 个要素
JSON Schema
网页工具

这里说明 OpenODC 的技术基础。不是重新发明一套分类，而是把标准的层级结构变成机器可读格式。

项目定位

OpenODC 做四件事

它不是认证，也不是排行榜。它是开放格式、公开样例、组合边界和网页工具的最小闭环。

01

统一格式

用同一套国标要素表达运行边界。

02

公开样例

基于公开资料反推，并标注来源置信度。

03

组合边界

把多要素叠加写成触发条件候选。

04

网页工具

查阅、对比、矩阵、编辑、厂家资料工作台。

这里修正一个关键误解：OpenODC 不只是把标准做成表格。真正有工程价值的是在表格上再加一层轻量组合边界。

单项表格的局限

单项允许，不等于组合可用

ODC 表格把真实场景拆成道路、天气、标线、目标物、速度和人员状态。但风险往往来自这些要素同时出现。

雨 / 雾 / 逆光
车道线磨损
施工改道
驾驶员分心
这不是四个独立格子，而是一个触发条件候选。

这是 OpenODC 的精髓之一。标准表格解决可比性，但真实安全问题不是单项要素线性相加。这里要把“组合边界”的必要性讲透。

Boundary combinations

OpenODC 增加了第二层边界

`boundary_combinations[]` 只记录公开资料可支撑的少量高价值组合。面向消费者，它说明“什么时候更该谨慎”；面向工程师，它是触发条件候选。

01组合条件

哪些 ODC 要素同时出现。

02关系判断

inside / outside / boundary / unknown。

03系统响应

抑制激活、触发退出、接管提醒或未披露。

04证据等级

官方、手册、运营规则、社区整理或公开缺口。

强调它是轻量第二层，不是完整 SOTIF 数据库。OpenODC 只收公开证据能支撑的组合，避免过度推断。

触发条件候选

从 ODC 到 SOTIF 的接口

组合边界不是安全结论，而是把公开 ODC 声明转成可讨论、可验证、可分配给测试和数据团队的触发条件候选。

OpenODC声明边界
DRIVEResearch真实暴露频次和参数分布
ROAM越界、异常和运营干预结果

这页把 OpenODC 放进你的整体方法论里：它不是 SOTIF 平台，但它给 SOTIF 分析提供可追溯的入口。

一份数据，多种入口

核心不是页面，而是同一份 JSON

Gallery、详情、Compare、Matrix、Editor、Workbench 都围绕同一套 ODC 数据模型展开，包括单项要素和组合边界。

这页适合对技术听众讲。项目真正的可复用资产不是某个页面，而是数据契约。

证据治理

证据比完整更重要

ODC 表不是越满越好。真正有价值的是每一项和每个组合都说明来自哪里，哪些是公开证据，哪些是推定，哪些没有公开说明。

手册明确 官方声明 社区整理 推定 公开资料未明确 结构性类别

这页强调 OpenODC 的方法论底线。不要把推定包装成官方声明，不要把公开资料覆盖说成厂家披露。

厂家直填

公开样例之外，还需要厂家版本

Workbench 的目标是让 OEM 或 Tier 1 在内部维护 ODC 和组合触发条件，SOP 或 OTA 时形成可审阅、可发布、可追溯的版本。

边界声明

OpenODC 不是智驾排行榜

它只回答一个更基础的问题：公开资料把一个功能的运行边界和组合触发条件讲清楚了多少。

不做能力排名

不替代安全认证

不把 L2 写成自动驾驶

不把推定写成官方披露

这页用于防御误读。对外传播时要反复强调，这是公开边界透明化工具，不是榜单。

AD 安全开源工具栈

边界、暴露、结果形成闭环

OpenODC 回答“声明的边界是什么”，DRIVEResearch 回答“这些组合在真实道路中多常见”，ROAM 回答“出了什么问题”。

把 OpenODC 放回你的整体开源布局里。它不是孤立项目，而是 AD safety stack 的设计边界端。
