VENDOR WORKBENCH

Vendor ODC Workbench

An authoring entry point for OEMs and Tier 1 suppliers: maintain operating boundaries, evidence links, and version context internally, then generate a reviewable OpenODC submission package.

Public Workbench — Drafts are stored in this browser only and are not uploaded to an OpenODC server. Public export creates a GitHub PR package that the submitter must review, sanitize, and submit for community review. Send feedback →
Local workspace: start from public samples where available, or create internal drafts.
Select a vendor identity to start.

What the Workbench Is For

Internal Management

  • One vendor often maintains multiple models, markets, automation levels, and feature variants.
  • ODC data needs to connect product, regulation, functional safety, SOTIF, testing, and after-sales traceability.
  • Each SOP or OTA should preserve the operating boundary, evidence sources, and review status at that time.
  • Before public release, teams need to separate publishable fields, internal fields, and items that still require verification.

Public Submission

  • Structured format: use the 144 GB/T 45312—2025 elements instead of free-form boundary prose.
  • Evidence binding: link key elements to manuals, official pages, operating rules, or regulatory material.
  • Version traceability: GitHub history records how public boundaries change over time.
  • Reviewable publication: PR review checks sources, terminology, and L2/L3/L4 semantics before merge.

Recommended Flow

  • Choose a vendor or Tier 1 identity and create the ADS feature record.
  • Open the editor and fill permitted conditions, limitations, and exit behavior across the 144 elements.
  • Bind key elements to official manuals, product pages, operating rules, or regulatory sources.
  • Generate a PR package at SOP, OTA, or public-release checkpoints.
  • Submit for GitHub review; after merge, the record appears in the OpenODC gallery.
