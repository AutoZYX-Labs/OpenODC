The real question

Not where it can drive

But where it should not be relied on.

What weather triggers exit?

What happens in work zones, flooding, or weak lane markings?

How much of this is public evidence?

This question lands with consumers, regulators, engineers, and media. Capability and boundary transparency are different things.

L2 / L3 / L4

Automation level is not an operating boundary

Two L2 systems can have very different road, weather, speed, and driver-monitoring boundaries. Level describes responsibility; ODC describes operating conditions.

L2Driver remains responsibleODC describes feature-use conditions
L3System takes DDT within limitsExit and takeover conditions must be explicit
L4Driverless within a domainGeofence, weather, and operation rules matter

Prevent the “ranking by automation level” misunderstanding. OpenODC addresses operating boundaries, not levels as such.

Plain-language ODC

ODC is a boundary list

It states the road, weather, object, digital, driver, and vehicle-state conditions under which a feature was designed to operate.

Road Infrastructure Objects Weather Digital info Personnel Vehicle state

Keep the definition intuitive first. The standard taxonomy comes next.

Positioning

OpenODC does four things

It is not certification and not a leaderboard. It is a minimum loop of open format, public samples, boundary combinations, and browser tools.

01

Format

Use one standard vocabulary for operating boundaries.

02

Samples

Extract public-source examples with explicit confidence labels.

03

Combinations

Turn multi-element boundaries into trigger-condition candidates.

04

Tools

Browse, compare, matrix, edit, and vendor intake workflows.

This corrects an important omission: OpenODC is not only a table. The engineering value comes from adding a lightweight combination-boundary layer on top of the table.

Limitation of element tables

Single-element permission does not mean combined-scene availability

ODC tables decompose real scenes into road, weather, lane markings, objects, speed, and driver state. Risk often emerges when these elements appear together.

Rain / fog / glare
Worn lane markings
Work-zone rerouting
Driver distraction
This is not four independent cells. It is one trigger-condition candidate.

This is one of the core ideas. The table gives comparability, but real safety issues are not linear sums of single ODC elements.

Boundary combinations

OpenODC adds a second boundary layer

`boundary_combinations[]` records a small number of public-source-supported, high-value combinations. For consumers, it says “when to be more cautious”; for engineers, it becomes trigger-condition candidates.

01Combined condition

Which ODC elements appear together.

02Relation

inside / outside / boundary / unknown.

03Expected response

Suppress activation, trigger exit, takeover prompt, or not disclosed.

04Evidence level

Official, manual, operating rule, curated, or public gap.

Stress that this is a deliberately small second layer, not a full SOTIF database. OpenODC only publishes combinations supported by public evidence.

Trigger-condition candidates

The interface from ODC to SOTIF

A boundary combination is not a safety conclusion. It translates a public ODC statement into a traceable candidate that testing, data, and safety teams can discuss and validate.

OpenODCDeclared boundary
DRIVEResearchReal exposure frequency and parameter distribution
ROAMBoundary exceedance, anomaly, and intervention outcome

Place OpenODC in the broader methodology: it is not a SOTIF platform, but it creates a traceable entry point for SOTIF analysis.

One data model, multiple entry points

The core asset is the JSON, not the page

Gallery, detail view, compare, matrix, editor, and workbench all render or author the same ODC document model, including single elements and boundary combinations.

For technical audiences, highlight the data contract as the reusable asset.

Evidence governance

Evidence matters more than completeness

A full-looking ODC table is not enough. Each item and each combination must say whether it comes from a manual, an official source, curation, inference, or a public gap.

Manual-backed Official statement Community curated Inferred Public-source gap Structural category

This is the core governance principle: do not package inference as vendor disclosure.

Vendor intake

Public samples are not enough

The Workbench points toward vendor-maintained ODC versions and combined trigger conditions: reviewable, publishable, and traceable across SOP and OTA changes.

Community extraction validates the method; long-term credibility needs vendor confirmation, institutional review, and version governance.

Boundary statement

OpenODC is not an AD ranking

It answers a narrower and more useful question: how clearly do public sources state a system's operating boundary and combined trigger conditions?

No capability ranking

No replacement for safety certification

No relabeling L2 as automation

No packaging inference as official disclosure

Use this slide to prevent misuse in public sharing.

Boundary, exposure, and outcome form a loop
