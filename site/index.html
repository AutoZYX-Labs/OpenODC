这是什么

OpenODC 是一个开源的 ODC（Operational Design Condition，设计运行条件）标准化定义、对比、查阅平台。它基于国家标准 GB/T 45312—2025《智能网联汽车 自动驾驶系统设计运行条件》，目标是衡量运行边界被公开资料说清楚了多少，而不是评价哪家智驾“更强”。

为什么要有这个

三个行业现状

  • OEM 各自定义 ODC，没有统一格式 —— 监管和第三方机构难以横向对比
  • 消费者看到的是功能名和营销话术，很难看到具体边界 —— 尤其容易混淆 L2 辅助驾驶与 L3/L4 自动驾驶
  • 国家标准 GB/T 45312—2025 给了完整层级结构，但缺少机器可读的对应物

OpenODC 的定位

不是要取代厂家自己的 ODC 文档系统。实际上，OEM 会在使用手册、车机教学里告诉用户"智驾什么时候建议使用、什么时候不应使用"，不少车企写得还挺详细。

但每家都用自己的写法、自己的分类、自己的术语，没有统一规范 —— 消费者跨品牌换车要重新学一遍，监管和第三方机构想做横向对比，也找不到统一的抓手。

OpenODC 提供一个统一的、机器可读的标准化格式 + 公开样例库，让设计运行条件可查、可比、可复用。当前样例均为社区基于公开资料提取，不代表厂家官方确认；只有标记为 vendor_confirmed 的记录才是厂家确认版本。

当前进展（v0.4.0 · Phase 0–4 MVP）

  • Phase 0：完整转录 GB/T 45312—2025（144 个元素 / 7 类）+ JSON Schema + TypeScript 类型
  • Phase 0：量化分级表机器可读化（风力、雨量、降雪、积雪、4 类能见度、光照）
  • Phase 1：公开样例库 Gallery（6 个典型样例，均标注公开资料覆盖与缺口）
  • Phase 1：Web 编辑器（层级树勾选 + 实时 JSON + 多格式导出）
  • Phase 2：双视图渲染（开发者 / 消费者，含来源、责任边界和公开资料覆盖标签）
  • Phase 2：多车型对比 Compare（diff 视图，2–4 个并排）
  • Phase 3：横向矩阵 Matrix（144 个国标要素 × 6 个样例）
  • Phase 3：GitHub PR 模板 + CI 自动校验（ajv-cli + element_id 引用检查）
  • Phase 4：厂家直填 Workbench MVP（内部管理、编辑器联动、生成 PR）
  • Apache 2.0 代码许可 + CC BY 4.0 数据许可
  • Phase 5：账号体系 + Supabase 后端 + 签名发布
  • Phase 5：与 ROAM 联动（事件 ↔ 涉事车型 ODC 自动关联）
  • Phase 5：ISO 34503 / BSI PAS 1883 / ASAM OpenODD 国际化映射

参与方式

对 OEM / AD 供应商

提交贵方某个车型 / 功能的官方 ODC 声明。我们会标记为 vendor_confirmed，作为该条目的厂家确认版本；也可选择只公开对外版本，保留内部细节。

对第三方机构 / 学术界 / 媒体

从公开材料（车主手册、运营规则、政府公告、媒体测评）提取 ODC 数据，标注来源置信度。提交 PR 后进入 community_reviewed 状态。
